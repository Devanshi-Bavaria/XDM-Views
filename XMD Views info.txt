XDM‑Views Project – Conversation Summary (LLM Context)
1. Project Goal

The project is about building Cross Data Model Views (XDM Views).

Goal: allow a single logical query/view to retrieve and combine data from multiple heterogeneous databases, specifically:

    Relational DB (RDBMS) – Customer table

    XML database – PurchaseOrder documents

The system should:

    Define metadata describing all databases and entities

    Define views using that metadata

    Build an execution engine that translates view definitions into:

        SQL queries for relational DB

        XPath queries for XML DB

    Merge results and return a tabular output

This is essentially a mini federated query engine.
2. Metadata Architecture

The system has three logical layers.
Meta‑Meta Schema (XSD)

Defines the structure of the metadata system.

It describes:

    Databases

    Entities

    Attributes

    Relationships

    Views

    Filters

    Projections

Purpose:
Validate MetaSchema.xml.
MetaSchema (metadata layer)

Describes the global schema across databases.

Example entities:

Customer (Relational)

customer_id
name
city

PurchaseOrder (XML)

order_id
customer_id
amount
item_name
item_category

XML entity definition includes:

BasePath = /PurchaseOrders/PurchaseOrder

Attributes map to XPath paths.

Example:

item_name → item/item_name
item_category → item/item_category

Full XPath is built as:

BasePath + attribute path

Example:

/PurchaseOrders/PurchaseOrder/item/item_name

Relationships

Example join:

Customer.customer_id = PurchaseOrder.customer_id

Stored as:

CustomerPurchaseJoin

Used by views to join datasets.
3. View Definitions

Views describe logical queries.

Example view:
HighValueCustomers

Logical query equivalent:

SELECT customer_id, name, city
FROM Customer, PurchaseOrder
WHERE Customer.customer_id = PurchaseOrder.customer_id
AND PurchaseOrder.amount > 10000

View XML includes:

    Projection

    BaseEntities

    RelationshipRef

    Filter

Example:

<View name="HighValueCustomers">
  Projection → customer attributes
  BaseEntities → Customer + PurchaseOrder
  RelationshipRef → CustomerPurchaseJoin
  Filter → amount > 10000
</View>

Other views:
CustomersByItem

customers who purchased Laptop

Filter:

PurchaseOrder.item_name = Laptop

CustomerPurchases

all purchases for customer_id = 5

4. XML Data Model

Example XML:

<PurchaseOrders>
  <PurchaseOrder>
    <order_id>101</order_id>
    <customer_id>5</customer_id>

    <item>
      <item_name>Laptop</item_name>
      <item_category>Electronics</item_category>
    </item>

    <amount>45000</amount>
  </PurchaseOrder>
</PurchaseOrders>

Important points:

    XML arrays may exist

    XPath naturally handles multiple elements

Example XPath:

/PurchaseOrders/PurchaseOrder[item/item_name='Laptop']

5. Query Translation

The execution engine converts view definitions into database‑specific queries.
Example translation

View filter:

PurchaseOrder.item_name = 'Laptop'

Using MetaSchema:

BasePath = /PurchaseOrders/PurchaseOrder
item_name path = item/item_name

Generated XPath:

/PurchaseOrders/PurchaseOrder[item/item_name='Laptop']

SQL generation

Customer IDs extracted from XML are used in SQL:

SELECT customer_id, name, city
FROM Customer
WHERE customer_id IN (...)

6. Execution Engine Design

The engine performs these steps:

1. Load MetaSchema.xml
2. Load views.xml
3. Resolve requested view
4. Identify entities and filters
5. Generate XPath for XML sources
6. Generate SQL for relational sources
7. Execute queries
8. Join results
9. Output result table

7. Engine Architecture

User Query (View Name)
        |
View Resolver
        |
Query Planner
     /      \
SQL Query   XPath Query
(RDBMS)       (XML)
     \      /
     Result Join
        |
     Output Table

8. Result Representation

Even though XML is hierarchical, the final output is flattened into rows.

Example output:

customer_id | name | item_name | category
-----------------------------------------
5           | John | Laptop    | Electronics
5           | John | Mouse     | Electronics

9. Key Design Decisions

Important architectural choices made:
1. Logical vs Physical Separation

Views remain logical.

Physical paths (XPath) are stored in MetaSchema.
2. BasePath Optimization

To avoid repeating long XPath prefixes:

BasePath = /PurchaseOrders/PurchaseOrder

Attributes store relative paths.
3. Federated Query Model

System behaves like a federated DBMS:

SQL → relational DB
XPath → XML DB
Join → execution engine
